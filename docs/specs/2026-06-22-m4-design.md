# M4 — TV data model + discovery (design)

> Milestone M4 of ROADMAP.md Part II. The deepest break from the movie pipeline's
> 1:1 `row = request = file` assumption, so the schema lands **first, behind monitoring
> flags**, leaving the validated movie loop untouched. M4 is **L** → split into sub-sessions.
> This doc records the locked decisions and the M5/M6 guardrails; it does not restate the
> roadmap.

## Session split

- **M4a (this session) — data layer.** series/seasons/episodes schema + migration, TMDB TV
  callbacks (+ Mox auto-cover), `Catalog.add_series_to_watchlist/2` persisting the tree with
  monitor flags. This *is* the M4 "Done when" test. **Shipped.**
- **M4b (next session) — discovery UX.** TV search in the grid, a series-detail LiveView with
  per-episode monitor toggles, the `"series"` PubSub topic (added when a subscriber exists), nav.

## Locked decisions

- **Separate `series` / `seasons` / `episodes` tables** (not polymorphic). *[roadmap-locked]*
- **Monitor-strategy default `:future`** for a newly added series (don't flood the client).
  Applied **uniformly** across seasons (specials included) — Sonarr-style specials carve-outs
  are an M6 monitoring concern, not M4.
- **TV add is admin-only direct** in M4 (`add_series_to_watchlist`, like `/settings`). No
  request/approval gate — and no leak, because there's no TV poller yet to auto-grab. The
  requester TV flow (series through the already-polymorphic `requests` table) lands with **M5**.
- **Episode = identity + monitoring only.** Council-driven cut from "copy all of Movie's pipeline
  fields": (a) reusing Movie's `status` (default `:requested`) is the roadmap's *named*
  approve-by-default trap, and (b) the season-pack **grab/download join table is locked for M5**,
  so `download_id`/`download_protocol` belong on the *grab*, not the episode. The "Done when" only
  needs monitor flags. Every deferred field is a clean additive M5 migration.
- **`tmdb_episode_id` is stored now** (free from `get_season`) — the stable key M6 reconciliation
  matches on across TMDB renumbering; backfilling it later would be a data migration. (By
  contrast `imdb_id` on series is *not* pulled forward — it's re-derivable in M5 at zero backfill
  cost, so it's dropped here; `tvdb_id` stays per the roadmap, populated from `external_ids`.)

## Schema (`priv/repo/migrations/20260622130000_create_tv_schema.exs`)

Additive; enums stored as `:string` per convention. `series` (tmdb_id uniq [un-named index →
column-derived `{:error, changeset}`, per the ecto_sqlite3 gotcha], tvdb_id, title, year,
poster_path, monitored, monitor_strategy). `seasons` (series_id FK + index, season_number,
monitored; uniq `[series_id, season_number]`). `episodes` (season_id FK + index, tmdb_episode_id,
episode_number, title, air_date, monitored; uniq `[season_id, episode_number]`).

The tree inserts in **one `Repo.insert`** via `Series.create_changeset/1` →
`cast_assoc(:seasons)` → `cast_assoc(:episodes)` (one transaction = one writer = WAL-correct).
Nested changesets do **not** `validate_required` their FK — `cast_assoc` fills it.

## TMDB (atomic: behaviour + HTTP impl)

`search_tv/1`, `get_series/1` (`append_to_response=external_ids` for `tvdb_id`; `imdb_id` for TV
is nested under `external_ids`, **not** top-level like movies), `get_season/2`. A total
`date_from/1` parser maps TMDB's `""`/missing `air_date` → `nil` **in the HTTP layer**, so the
`:date` cast and the `>= today` comparison never see a bad string. Monitor flags use
`Date.utc_today/0`; tests use far-past/far-future/`nil` dates so they never depend on the wall
clock.

## Test-infra fix (carried by M4a, applies suite-wide)

`config/test.exs` `pool_size: 5 → 1`. The Ecto SQL Sandbox wraps each test in a *deferred*
transaction; a read-then-write (`find_or_create`-style) test that races a concurrent writer hits
`SQLITE_BUSY` on snapshot conflict, which `busy_timeout` **cannot** rescue (it only helps
`BEGIN IMMEDIATE` — see `repo_concurrency_test.exs`). This was a latent ~3% flake at baseline; M4a's
heavy `cast_assoc` writes amplified it to ~20%. One sandbox connection serializes writers (SQLite
is single-writer anyway), making the suite deterministic at no speed cost (still ~1s). Does not
mask any production path — prod uses real connections + `busy_timeout`, proven in M0.

## Guardrails for M5/M6 (the reason this doc exists)

- **M5 TV poller MUST select eligible episodes as `monitored AND file_path IS NULL AND
  air_date <= today`** — never a bare `status == :requested` sweep (which would grab the entire
  back-catalogue: the roadmap's approve-by-default top-risk). Episode `status` will be added in M5
  as pure pipeline-progress.
- **M5 grab/download join table owns `download_id`/`download_protocol`** (one download → N
  episodes); the episode owns its post-fan-out `file_path` + import result. Don't put the grab's
  download id back on the episode.
- **M5 `add_series_to_watchlist` is 1 + N synchronous TMDB calls** (one `get_series` + one
  `get_season` per season). Fine for M4a (mocked). When M4b calls it from a LiveView, run it off
  the LiveView process (Task) or show a loading state — a 20-season show is ~21 round-trips.
- **M6 wanted-episodes query** is a targeted `WHERE` (not load-all-filter). Add a
  `[:monitored, :air_date]` index only if a real instance shows a slow scan — matching the movie
  poller's no-status-index, household-scale precedent.
- **M6 reconciliation** matches on `tmdb_episode_id` (renumbering-safe), not `(season, episode)`
  position.
