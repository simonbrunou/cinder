# M6 Design — TV monitoring sweep + TMDB refresh + calendar

**Milestone:** M6 (ROADMAP Part II). Size M. Branch `m6-tv-monitoring-sweep-calendar`.
**Date:** 2026-06-22.

## Goal (from ROADMAP)

Close the Sonarr monitoring loop: a wanted-episodes query drives the search sweep
efficiently, and an air-date-aware refresh marks newly-aired monitored episodes
search-eligible so they grab automatically. Leanest cut: **poll TMDB**, not per-tracker RSS.

**Done when:** conventions pass + a test proves a monitored, just-aired episode becomes
search-eligible and grabs automatically, and the wanted-episodes query is used (not a full scan).

## Reframing — what already exists

A read of the current TV subsystem (M4a/M4b/M5a/M5b/M5c) shows three of the five
milestone build-items are already landed:

- **Air-date eligibility already works.** `Catalog.wanted_episodes/0` (`catalog.ex:532`)
  already filters `monitored AND file_path IS NULL AND grab_id IS NULL AND air_date IS NOT NULL
  AND air_date <= today`, joined to `seasons` on `season_number > 0`. With the derived-state
  episode model (no status enum), "a just-aired episode becomes search-eligible" happens
  **automatically as time passes** — there is no flag to flip. The Done-when is already true for
  any episode whose `air_date` was correct at add time.
- **Monitor-strategy is already enforced** at the per-episode `monitored` leaf, applied once at
  add via `monitored?/3` (`catalog.ex:255`); the sweep reads `e.monitored`.
- **The wanted-driven sweep already exists** in `TvPoller.search_wanted/1` — it calls
  `wanted_episodes/0` (does not scan every episode row), backs off on `search_attempts`, groups
  by `{series, season}`, and grabs via `Acquisition.best_releases`.

So M6's real work is narrow: **keep the `air_date`/tree data fresh so the existing eligibility
fires on time**, add the index that query needs, and add the calendar view. The single most
important Done-when case is the **late air-date fill**: an episode added with `air_date: nil`
(monitored under `:future`) is invisible to `wanted_episodes/0` until a refresh fills the date.

## Decisions (locked)

- **Scope:** all three pieces this session, one PR.
- **Refresh depth:** *fresh + grow* — update existing episodes in place by `tmdb_episode_id`,
  AND insert newly-announced episodes + new seasons (Sonarr-correct: a show's next season
  auto-appears and gets grabbed). Don't delete vanished rows.
- **Scheduling:** a dedicated `Cinder.Catalog.Refresher` GenServer mirroring the poller skeleton,
  `:start_poller`-gated, 12h interval as module config (not a `/settings` field — no int-coercion
  seam exists there and adding one isn't justified by one interval). No Oban (roadmap-locked).
- **Refresh scope per tick:** all monitored series.
- **New episodes from a refresh:** apply the series' `monitor_strategy` via `monitored?/3`.
- **Vanished episodes/seasons:** left untouched (never drop a row that may hold a `file_path`).
- **Specials (`season_number == 0`):** stay excluded from `wanted_episodes/0` (parser/scorer
  can't address `{Season:0}` — already deferred). Refresh still processes season 0 for tree
  completeness; those rows are harmlessly never searched.

## Piece 1 — Wanted-episodes index (S)

`wanted_episodes/0` has no supporting index (episode indexes today: `season_id`,
`(season_id, episode_number)` unique, `grab_id`).

- **New migration** adds a partial index on `episodes(air_date)` with predicate
  `WHERE file_path IS NULL AND grab_id IS NULL AND monitored` — the wanted set is a small slice of
  the table, so a partial index is the right tool and is exactly what the planner wants for the
  `air_date <= today` range scan. (SQLite stores booleans as 0/1; the predicate is written in the
  form the adapter emits — verified empirically during impl.)
- **Verification:** a test builds the wanted query's SQL via `Ecto.Adapters.SQL.to_sql(:all, Repo, q)`,
  runs `EXPLAIN QUERY PLAN <sql>` through `Repo.query!`, and asserts `episodes` is accessed via an
  index (no full `SCAN episodes`).
  - *Risk:* SQLite's planner can prefer a scan on tiny tables. If the EXPLAIN assertion proves
    planner-flaky at test scale, fall back to seeding enough non-matching rows and/or asserting the
    index exists in `sqlite_master`. The exact assertion is nailed empirically during impl; the
    index itself is correct regardless.

## Piece 2 — TMDB refresher + reconciliation (the core; carries Done-when)

### `Cinder.Catalog.refresh_series/1`

Re-fetch one series from TMDB and reconcile against stored rows, in **one `Repo.transaction`**
(WAL + `busy_timeout` discipline; one writer per series), broadcasting `{:series_updated, id}`
once via the existing `broadcast_series/1` on the `"series"` topic.

1. `tmdb().get_series(tmdb_id)` → season-number list; `tmdb().get_season(tmdb_id, n)` per season.
   The behaviour callbacks already return `tmdb_episode_id`, `episode_number`, `title`, `air_date`
   (a `Date` or `nil`) — no behaviour change needed.
2. **Match each TMDB episode to a stored row by `tmdb_episode_id`** (series-wide, not just within a
   season — renumbering can move an episode across seasons):
   - **Found** → update `air_date`, `episode_number`, `title`, `season_id` in place. **Preserve**
     `monitored`, `file_path`, `grab_id`, `search_attempts`, `import_attempts`. *(Late-air-date fill
     + renumber fix.)*
   - **Not found** → insert as a new episode under its season with
     `monitored: monitored?(strategy, air_date, today)` — re-applies the series' strategy.
3. **New seasons** (a `season_number` with no stored season) → insert the season
   (`monitored: strategy != :none`), then its episodes (per step 2's not-found branch).
4. **Vanished rows** (stored but absent from TMDB) → left untouched.
5. Writes are plain changesets — `Repo.update` / `Repo.insert` with
   `unique_constraint([:season_id, :episode_number])` so a collision returns `{:error, changeset}`
   rather than raising — **not** `transition_episode/2` (that choke-point is for
   `file_path`/`grab_id`/counters; identity + monitoring are not pipeline state, like
   `set_episode_monitored/2`). Per the `[[ecto-sqlite3-constraint-name]]` rule the unique index is
   column-derived (no custom name), so `unique_constraint([:season_id, :episode_number])` resolves
   to the right name.

**Renumber-collision ceiling (ponytail):** a true mid-season reorder can transiently collide on
`(season_id, episode_number)`. The common cases (late-date fill, appended episode, new season, title
change) never collide, so they always commit. A collision's handling depends on Ecto wrapping each
constraint-bearing write in a savepoint inside the transaction (verify `ecto_sqlite3` supports
`SAVEPOINT` — to confirm empirically): if it does, the colliding episode's number-update returns a
changeset error → logged and skipped while the rest of the series commits; if it doesn't, the
collision rolls back that one series' refresh and it retries next cycle. **Both outcomes are
acceptable** — the collision self-heals over cycles as the colliding row also moves, or needs a
manual fix. Comment names the upgrade: two-pass renumber if it ever matters. The Done-when case
(late-date fill) is collision-free either way.

`refresh_series/1` returns `{:ok, series}` or `{:error, reason}` (TMDB fetch failure short-circuits
before any write, mirroring `create_series`/`fetch_seasons`).

### `Cinder.Catalog.Refresher` (GenServer)

Exact copy of the poller skeleton: `init → {:continue, :schedule} → handle_continue → schedule/1 →
Process.send_after`, plus a sync `poll/0` `handle_call` for tests and an `isolate/2` (rescue **and**
catch, like `TvPoller.isolate`) so one series' TMDB/DB failure skips that series, not the tick.

- `@default_interval` 12h; `config_interval/0` reads `Application.get_env(:cinder, __MODULE__)`
  `[:interval]` (module-config override, like the pollers' `@default_interval`).
- Each tick: `Catalog.list_series()` → filter `& &1.monitored` → `isolate` + `refresh_series`.
- Added to `Cinder.Application.poller_child/0` under the existing `:start_poller` gate, so
  `config/test.exs`'s `start_poller: false` keeps it from auto-running in tests (tests call
  `refresh_series/1` / `Refresher.poll/0` directly).
- First tick fires after one interval (mirrors the pollers) — the add already populated fresh data,
  so no immediate run is needed.

### Touch

`add_series_to_watchlist/2`'s early-return-for-existing is **unchanged**; the periodic refresher is
the re-sync path. Its docstring ("no re-sync (that's M6)") gets a one-line update pointing at
`refresh_series/1`.

## Piece 3 — Calendar / upcoming view (M)

- **`Catalog.upcoming_episodes/0`** — monitored episodes with an `air_date` in a window
  (`today - 7 .. today + 90`), ordered ascending by `air_date`, `preload: [season: :series]`. Pure
  read; no pipeline writes.
- **`CinderWeb.CalendarLive`** at `/calendar`, in the `:admin` `live_session` (matches `/series`),
  subscribes to the `"series"` topic and re-queries on `{:series_updated, _}` so badges stay live.
  Renders a daisyUI table ordered by date with a derived state badge per episode:
  Available (`file_path`) / Downloading (`grab_id`) / Wanted (aired + monitored + missing) /
  Upcoming (future air_date). Reuses SeriesDetailLive's episode-state rendering if a shared helper
  exists; otherwise a small inline helper.
- **Nav:** a `/calendar` link in the admin block of `root.html.heex` and the TV nav, role-gated like
  the existing `/series` link.

## Tests (Done-when + regression)

1. **Late air-date → auto-grab (the Done-when):** episode added `air_date: nil`, monitored under
   `:future`; mock TMDB returns it with a past `air_date`; `refresh_series` fills it;
   `wanted_episodes/0` now includes it; the TvPoller search pass (mocked indexer + client) creates a
   grab linking it.
2. **Index used:** EXPLAIN QUERY PLAN of the wanted query shows `episodes` is not a full scan.
3. **Reconciliation:** match-by-`tmdb_episode_id` updates an episode in place (renumber: number/season
   change, no orphan/duplicate, `monitored`/`file_path` preserved); a genuinely new episode is
   inserted with strategy applied; a new season is inserted; a vanished row is left untouched.
4. **Calendar:** `upcoming_episodes/0` returns monitored windowed episodes in air-date order; the
   LiveView renders them with correct badges.
5. **Movie loop untouched:** the existing suite stays green (regression guard).

## Out of scope (deferred)

- Specials grabbing (parser/scorer limitation — `{Season:0}`).
- Per-episode TV size-band as a `/settings` field (M8; no settings int-coercion seam yet).
- Vanished-row deletion / un-monitoring (leave-untouched is the v1.0 stance).
- Changing `add_series_to_watchlist`'s re-add behaviour (refresher is the re-sync path).
